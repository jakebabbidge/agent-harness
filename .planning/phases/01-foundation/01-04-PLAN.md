<feature>
  <name>Container Lifecycle Manager</name>
  <files>src/container/manager.ts, src/container/registry.ts, src/container/manager.test.ts</files>
  <behavior>
    CONT-01 (container lifecycle):
    - ContainerManager.createContainer(taskId, repoPath) → creates container, starts it, registers in registry, returns ContainerInfo
    - ContainerManager.stopContainer(taskId) → stops and waits for removal; unregisters from registry
    - ContainerManager.reclaimOrphans() → lists all containers with label `agent-harness=true`, force-removes them; called on startup
    - Container name follows pattern: `agent-harness-task-<taskId>`
    - Container labels: `{ 'agent-harness': 'true', 'agent-harness.task-id': taskId }`
    - `AutoRemove: true` in HostConfig — Docker daemon removes container; harness does not need to call `remove()` explicitly after `stop()`

    CONT-02 (isolation):
    - `NetworkMode: 'none'` in HostConfig — no network access
    - `ReadonlyRootfs: true` in HostConfig — root filesystem is read-only
    - `Tmpfs: { '/tmp': '', '/home': '' }` in HostConfig — writable scratch paths
    - `Binds: ['<repoPath>:/workspace:rw']` — repo mounted at /workspace

    CONT-01 + CONT-02 validation via inspect:
    - After createContainer, calling `docker.getContainer(id).inspect()` returns HostConfig matching expected isolation flags
    - `NetworkMode` is `'none'`, `AutoRemove` is `true`, `ReadonlyRootfs` is `true`
  </behavior>
  <implementation>
    `src/container/manager.ts`:
    - `ContainerManager` class takes a `Dockerode` instance in constructor (enables testing with real or mocked docker).
    - Default export factory: `createContainerManager(socketPath?: string): ContainerManager` creates with `new Dockerode({ socketPath: socketPath ?? '/var/run/docker.sock' })`.
    - Use `node:20-alpine` as the default image (lightweight, available on Docker Hub).
    - `createContainer` calls `docker.createContainer(...)` then `container.start()`, then registers to registry.
    - `stopContainer` calls `container.stop({ t: 5 })` — 5 second grace period. Because AutoRemove is true, do NOT call `container.remove()` afterward — it will 404.
    - `reclaimOrphans` uses `docker.listContainers({ all: true, filters: { label: ['agent-harness=true'] } })` then `remove({ force: true })` on each.
    - All relative imports use `.js` extension.

    `src/container/registry.ts`:
    - `ContainerRegistry` class — in-memory `Map<TaskId, ContainerInfo>`.
    - Methods: `register(info: ContainerInfo)`, `get(taskId: TaskId): ContainerInfo | undefined`, `getAll(): ContainerInfo[]`, `unregister(taskId: TaskId)`.

    `src/container/manager.test.ts`:
    - Mark tests with `// @integration — requires Docker socket` comment at top of file.
    - Use `describe.skipIf(!process.env.DOCKER_AVAILABLE)` or check socket existence: `import { existsSync } from 'fs'; const hasDocker = existsSync('/var/run/docker.sock');` and `describe.skipIf(!hasDocker)('ContainerManager integration', ...)`.
    - This lets CI without Docker skip rather than fail.
    - Test createContainer: create a container, inspect it, verify HostConfig fields.
    - Test stopContainer: verify container is gone after stop (inspect throws or returns 404).
    - Test reclaimOrphans: create a container manually via dockerode (bypassing manager), call reclaimOrphans, verify it is removed.
    - Use `afterEach` to clean up any containers that survived a failed test.
    - Pull `node:20-alpine` image once — it may already be present; use `docker.pull()` only if not found, or rely on it being pre-pulled in the test environment. Prefer letting docker pull implicitly on createContainer.

    **Note:** SIGKILL-safe cleanup is a MANUAL-ONLY verification per the validation strategy — it is NOT tested in automated tests. The test coverage for CONT-01 is create/start/stop/inspect only.
  </implementation>
</feature>
