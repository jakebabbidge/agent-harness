  <name>Task 1: Replace sequential runner with DAG-based concurrent runner</name>
  <files>src/workflow/runner.ts, src/workflow/runner.test.ts</files>
  <behavior>
    Concurrent execution (WKFL-03):
    - Two independent nodes (no edges between them) execute concurrently via Promise.allSettled
    - Verify concurrency: mock executor with delays, measure total time is less than sum of individual times
    - Each node gets a unique runId (UUID)
    - Each node's template is rendered before execution

    Conditional routing (WKFL-04):
    - After a node completes, outgoing edges are evaluated using evaluateCondition
    - Only downstream nodes whose incoming edge conditions ALL pass are scheduled
    - If a node has multiple outgoing edges, only matching ones activate
    - Unconditional edges always activate
    - Node whose incoming conditional edge fails is marked 'skipped' in state
    - resultText is parsed as JSON for condition evaluation; if not valid JSON, treat entire resultText as a string under key "raw"

    State persistence (WKFL-05):
    - State is initialized with all nodes as 'pending' at run start
    - Each node marked 'running' before execution, 'completed'/'failed' after
    - State is saved via saveRunState after each node status change
    - On resume: runWorkflow accepts optional existingState parameter
    - When existingState provided, completed nodes are skipped, 'running' nodes treated as 'pending' (re-run)

    Error handling:
    - Single node failure does NOT stop independent branches (unlike sequential runner)
    - If all paths to a downstream node include a failed node, that downstream node is skipped
    - WorkflowResult.status is 'success' only if all non-skipped nodes completed successfully

    Signature change:
    - `runWorkflow(workflow, executor, options?: { runId?: string; state?: WorkflowRunState; workflowPath?: string })` -- backward compatible, options is optional
    - When no options provided, generates a fresh runId and creates initial state
  </behavior>
  <action>
    1. Rewrite src/workflow/runner.ts:
       - Import topologicalTiers from dag.ts, evaluateCondition from condition.ts, saveRunState/loadRunState from state.ts
       - Keep WorkflowResult export interface (add 'skippedNodeIds?: string[]' field)
       - New signature: `runWorkflow(workflow: WorkflowDef, executor: TaskExecutor, options?: { runId?: string; state?: WorkflowRunState; workflowPath?: string }): Promise<WorkflowResult>`
       - Implementation flow:
         a. Generate runId (from options or uuidv4)
         b. Initialize state (from options.state or fresh with all nodes 'pending'; if resuming, treat 'running' as 'pending')
         c. Build completed set from state
         d. Compute active edges: start with all edges, filter by condition evaluation as upstream nodes complete
         e. Main loop: compute ready nodes (all upstream deps completed AND all incoming edge conditions met), execute with Promise.allSettled, update state, save state, repeat until no more ready nodes
         f. For condition evaluation: parse node's resultText as JSON. If parse fails, wrap in `{ raw: resultText }`. Pass parsed object to evaluateCondition.
         g. After loop: determine overall status (success if all non-skipped nodes completed)
       - Use renderTemplate for each node before executor.executeTask (same as current runner)

    2. Rewrite src/workflow/runner.test.ts:
       - Mock renderTemplate (same pattern as existing tests)
       - Mock executor with configurable per-node results

       - Mock state.ts's file I/O only (use createStateManager with a tmpdir)
       - Tests:
         a. "single node workflow executes and returns success" (backward compat)
         b. "two independent nodes execute concurrently" -- use timing: mock executor adds 50ms delay, verify total time < 150ms (sequential would be 100ms+)
         c. "linear chain A->B executes in order" -- verify B gets called after A
         d. "diamond DAG executes correctly" -- A->[B,C]->D, verify D runs after both B and C
         e. "conditional edge: matching condition activates downstream" -- node A outputs `{"status":"ok"}`, edge A->B has condition equals "ok", B executes
         f. "conditional edge: non-matching condition skips downstream" -- node A outputs `{"status":"bad"}`, edge A->B has condition equals "ok", B is skipped
         g. "resume skips completed nodes" -- provide state with node A completed, verify A's executor is not called, B runs
         h. "resume treats 'running' nodes as pending" -- provide state with node A as 'running', verify A is re-executed
         i. "node failure does not block independent branch" -- A fails, B (independent) still runs
