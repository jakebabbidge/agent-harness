**Rewrite `src/executor/executor.ts`:**

Remove the SDK import (`@anthropic-ai/claude-agent-sdk`) and QuestionStore dependency. The new TaskExecutor delegates to ContainerManager instead.

New constructor signature:
```typescript
constructor(containerManager: ContainerManager)
```

New `executeTask` implementation:
1. Create `.harness/` directory in worktreePath: `await fs.mkdir(path.join(worktreePath, '.harness'), { recursive: true })`
2. Write prompt to file: `await fs.writeFile(path.join(worktreePath, '.harness/prompt.txt'), prompt, 'utf-8')`
3. Generate a container taskId from runId: use `runId` directly as the taskId (or `runId.slice(0, 12)` if needed for Docker name constraints -- container names must be valid Docker names)
4. Create container: `await this.containerManager.createContainer(taskId, worktreePath, '.harness/prompt.txt')`
5. Wait for exit: `const { StatusCode } = await this.containerManager.waitForExit(taskId)`
6. Read RESULT.md from worktree (same pattern as current -- try/catch for ENOENT, return empty string if missing)
7. Return `{ exitCode: StatusCode, resultText }`

Important: The container is already started by `createContainer`. `waitForExit` blocks until the Claude CLI process inside the container finishes. After wait returns, AutoRemove cleans up the container. We read RESULT.md from the host filesystem (worktree mount persists).

Remove: All SDK-related code (query, canUseTool, async generator iteration), QuestionStore dependency, purgeRunDir call.

Note on HITL: Per the research, HITL in the container model is an open question. For v1, the agent runs non-interactively with `--dangerously-skip-permissions`. The QuestionStore/HITL path is not wired for containerized execution. The old HITL code is removed from TaskExecutor -- it can be re-added when the container HITL mechanism is designed.

Import `ContainerManager` type from `'../container/manager.js'`.

The runner does NOT manage container lifecycle directly -- that is now handled inside `TaskExecutor.executeTask()`. The runner's responsibility is:
1. Create the worktree (already done)
2. Render the template (already done)
3. Call `executor.executeTask(rendered, worktreePath, nodeRunId)` (already done)
4. Clean up worktree (already done)

Since TaskExecutor now takes ContainerManager in its constructor, the runner doesn't need to change its execution loop. The ContainerManager is passed to TaskExecutor at construction time (in CLI), not at each node execution.

However, we need to verify the runner passes the correct `worktreePath` (the isolated worktree, not the original repo). Looking at the current code, `executionRepo` is already set to `worktreeInfo.worktreePath` when a worktree is created. This is correct -- the container will mount this worktree path.

The only change needed in runner.ts is adding `containerManager` to the options interface for future extensibility and documentation purposes. The actual wiring happens at the CLI level where TaskExecutor is constructed with the ContainerManager.

If no other runner changes are needed after reviewing the flow, document this in the summary -- the runner already delegates to executor correctly.

**Update `src/cli/run.ts`:**

Add imports:
```typescript

```

Remove `QuestionStore` import (no longer needed by TaskExecutor).

In `runCommand`, before any execution:
1. Create ContainerManager: `const containerManager = createContainerManager();`
2. Reclaim orphans from previous crashed runs: `await containerManager.reclaimOrphans();`
3. Ensure image exists: `await ensureImage(containerManager['docker']);` -- or expose docker on the class. Better: add a `getDocker()` method to ContainerManager, or pass the Dockerode instance separately. Simplest approach: `ensureImage` takes no args and uses `docker build` CLI directly (no Dockerode needed for build). Adjust `ensureImage` in image.ts if needed -- it can check via `docker images -q agent-harness:latest` and build via `docker build`.

Actually, reviewing the design: `ensureImage` should be self-contained. Change its signature to `ensureImage(): Promise<void>` (no Dockerode param). It uses `child_process.execFile` to run `docker images -q agent-harness:latest` for the check and `docker build` for the build. This avoids exposing Dockerode internals.

Update TaskExecutor construction:
```typescript
// OLD: const executor = new TaskExecutor(questionStore);
const executor = new TaskExecutor(containerManager);
```

Remove `QuestionStore` creation since TaskExecutor no longer needs it.

Remove the commented-out API key check (no longer needed -- auth is via mounted ~/.claude).

Log: `console.log('[agent-harness] Reclaiming orphaned containers...');` before reclaimOrphans.

Both workflow mode and template mode use the same executor, so both paths are covered.
